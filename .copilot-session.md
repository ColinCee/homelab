## Issue #59: Worktree lifecycle: deferred cleanup with retention period

## Problem

Three issues prevent worktree debugging and block the unified implement lifecycle (#54):

1. **`/reviews` is not a named volume** — container restarts wipe all worktrees, session transcripts, review JSON, and CLI state.
2. **Eager deletion** — `cleanup_worktree()` in `finally` blocks deletes worktrees immediately after each task. No retention for debugging.
3. **`--resume` impossible** — the CLI's `.copilot/` directory (session state) is inside the worktree and gets deleted. This blocks the unified implement lifecycle's fix loop.

## What gets lost today

| Artifact | Location | Read into memory? | Persisted externally? |
|----------|----------|-------------------|----------------------|
| Session transcript | `.copilot-session.md` | ✅ into CLIResult | Logged at DEBUG only |
| Review JSON | `.copilot-review.json` | ✅ parsed → posted to GitHub | Raw file lost |
| CLI session state | `.copilot/` | ❌ never read | **Gone — blocks --resume** |
| Code changes | worktree files | ❌ | Pushed to GitHub (if successful) |

## Required changes

### 1. Add `/reviews` named volume (compose.yaml)

```yaml
volumes:
  repo-cache:
  reviews:

services:
  agent:
    volumes:
      - repo-cache:/repo.git
      - reviews:/reviews    # NEW
```

### 2. Deferred cleanup with retention

- Write a `.cleanup-after` marker file (expiry timestamp + branch name) instead of deleting
- Add `reap_old_worktrees()` — scans `/reviews/`, deletes worktrees past expiry
- Call reaper on startup (FastAPI lifespan) and opportunistically
- `WORKTREE_RETENTION_SECONDS = 14 * 86400` (14 days), configurable via env

### 3. Handle re-reviews

`create_worktree()` currently deletes existing worktrees before re-creating (needs fresh PR head). Options:
- **Replace** (current) — latest worktree only, simpler
- **Version** (`pr-42-v1`, `pr-42-v2`) — keeps history, more disk

Replace is probably fine — the value is debugging the *latest* run.

## Dependency

This is a prerequisite for the unified implement lifecycle (#54) — `--resume` needs the `.copilot/` state to survive across CLI sessions within a single task.

1. # ADR-004: Isolated Code Review Agent
2. 
3. **Date:** 2026-04-07
4. **Status:** Accepted
5. 
6. ## Context
7. 
8. The homelab has a self-hosted AI code review agent running on the Beelink. The current implementation (PR #27) has fundamental limitations:
9. 
10. 1. **No codebase context** — the agent only receives the PR diff text. It can't check callers, types, imports, or project conventions. It's essentially a glorified linter.
11. 2. **No credential isolation** — the agent container mounts Colin's `~/.config/gh` directory, giving it full access to GitHub (push, merge, delete repos, bypass branch protection).
12. 3. **No network isolation** — the container can reach any service on the local network (Dokploy, other containers, host SSH).
13. 4. **No filesystem isolation** — host directories are mounted read-only but still accessible.
14. 
15. These issues mean any prompt injection or model misbehaviour could have real consequences — merging PRs, modifying repos, or accessing local services.
16. 
17. ## Options Considered
18. 
19. ### Option A: `gh pr edit --add-reviewer @copilot`
20. 
21. Use GitHub's built-in Copilot code review. One CLI command, zero infrastructure.
22. 
23. - ✅ Full codebase context via GitHub's infrastructure
24. - ✅ Zero maintenance
25. - ❌ No control over review prompt, severity model, or output format
26. - ❌ No self-hosted — depends entirely on GitHub's service
27. - **Verdict:** Too limited for customisation needs. Good fallback.
28. 
29. ### Option B: Copilot CLI headless in isolated container
30. 
31. Run Copilot CLI in non-interactive mode (`-p` flag) inside a locked-down Docker container. The CLI has full tool use (grep, read files, search codebase) but the container has no access to the host or Colin's credentials.
32. 
33. - ✅ Full codebase context (grep, file reading, LSP)
34. - ✅ Reads `.github/copilot-instructions.md` automatically
35. - ✅ Standalone binary — no Node.js runtime needed
36. - ✅ JSON output (`--output-format json`) for structured parsing
37. - ✅ Credential isolation via fine-grained PAT + GitHub App
38. - ✅ Parallel reviews via git worktrees
39. - ❌ More infrastructure to maintain
40. - **Verdict: Chosen.** Best balance of capability and security.
41. 
42. ### Option C: Enrich current agent with file context
43. 
44. Keep the FastAPI + direct Copilot API approach but clone the repo and send full file contents alongside the diff.
45. 
46. - ✅ Simple enhancement to existing code
47. - ❌ Still no tool use — model can't decide what context it needs
48. - ❌ Doesn't solve credential/network isolation
49. - **Verdict:** Insufficient. Doesn't address the core problems.

68.     → Parse output → Post review via GitHub App token

72. ## Isolation Model
73. 
74. ### Credentials (principle of least privilege)
75. 
76. | Credential | Purpose | Scope |
77. |---|---|---|
78. | `COPILOT_GITHUB_TOKEN` (fine-grained PAT) | Copilot CLI LLM access | "Copilot Requests" permission only |
79. | GitHub App private key | Post reviews, read PR contents | Pull Requests (write), Contents (read) |
80. 
81. What the container does NOT get:
82. - ❌ Colin's `~/.config/gh` — full GitHub auth never enters the container
83. - ❌ Docker socket — can't manage other containers
84. - ❌ `--privileged` flag — standard container sandbox
85. - ❌ Host directory mounts — no access to host filesystem
86. 
87. ### Network
88. 
89. The container runs on the default Docker bridge with unrestricted outbound access. The security boundary is credential scoping and trigger gating, not the network layer.
90. 
91. **Why no egress proxy:** An HTTP proxy (Squid) was considered but dropped:
92. - Docker's `internal: true` networks don't support port publishing
93. - Proxy env vars are advisory-only — any tool can bypass `HTTP_PROXY`
94. - Data can be exfiltrated through allowed API endpoints (create gists, post issues)
95. - DNS exfiltration bypasses HTTP proxies entirely
96. - Adds operational complexity for marginal security gain
97. 
98. **Threat model:** The realistic attack is prompt injection via malicious PR content causing the CLI to exfiltrate credentials. Mitigations:
99. - Fork PRs are blocked from triggering reviews (checked in workflow + agent)
100. - `/review` command is role-gated to OWNER/MEMBER/COLLABORATOR
101. - Credentials are scoped: PAT has only Copilot Requests, App has only PR write + contents read
102. - No host filesystem, Docker socket, or Colin's personal credentials in the container
103. 
104. This matches industry practice — GitHub Copilot code review, CodeRabbit, Qodo, and similar tools all run with repo access and credentials without egress proxies. The trust boundary is who can trigger reviews, not what network the agent can reach.
105.

109. - Worktrees in `/reviews/` — ephemeral, wiped on restart

118. cpus: 2.0
119. ```

29. - ❌ `internal: true` networks don't support port publishing — can't receive inbound from Tailscale on `100.100.146.119:8585`

31. - ❌ HTTP proxy env vars (`HTTP_PROXY`) are advisory — `git`, `curl`, and the Copilot CLI binary can ignore them
32. - ❌ DNS exfiltration bypasses HTTP proxies entirely
33. - ❌ Three containers (agent + proxy + ingress) instead of one
34. - **Verdict:** Rejected. The port publishing constraint is a hard blocker, and the proxy is bypassable.
35. 
36. ### Option B: iptables/nftables egress rules on the host
37. 
38. Add firewall rules to restrict the agent container's outbound to specific IPs/CIDRs for GitHub and Copilot endpoints.
39. 
40. - ✅ Enforced at the kernel level — can't be bypassed from inside the container
41. - ❌ GitHub API uses dynamic IPs across multiple CDN ranges — requires polling the [GitHub meta API](https://api.github.com/meta) and updating rules
42. - ❌ Copilot API endpoints aren't published — would need to discover and maintain the IP list
43. - ❌ Rules break silently when IPs rotate, causing hard-to-debug failures
44. - ❌ Fragile — ties operational reliability to IP stability of third-party services
45. - **Verdict:** Rejected. IP-based filtering is too fragile for services with dynamic infrastructure.
46. 
47. ### Option C: Split into orchestrator + CLI containers
48. 
49. Separate the orchestrator (FastAPI, credentials, GitHub API access) from the CLI (Copilot binary, minimal network). Communicate via shared volume.
50. 
51. - ✅ CLI container could run on a restricted network (only Copilot API)
52. - ✅ Credential isolation — CLI never sees the App private key
53. - ❌ Review CLI still needs `GH_TOKEN` to post reviews via `gh` — still has GitHub API access
54. - ❌ Doubles container count and operational complexity
55. - ❌ Shared volume coordination adds failure modes (race conditions, stale state)
56. - ❌ Marginal gain — the implement CLI already gets no `GH_TOKEN`
57. - **Verdict:** Rejected. Significant complexity for minimal security gain. The implement path is already credential-isolated.
58. 
59. ### Option D: Accept unrestricted outbound, rely on existing controls
60. 
61. Keep the current model. The security boundary is credential scoping and trigger gating, not the network layer.
62. 
63. - ✅ Simple — one container, standard Docker networking
64. - ✅ Existing mitigations cover the realistic threat model (see below)
65. - ✅ Matches industry practice (GitHub Copilot, CodeRabbit, Qodo all run without egress filtering)
66. - ❌ A compromised CLI process could reach any internet host
67. - **Verdict: Chosen.** The marginal security of egress filtering doesn't justify the complexity or fragility.
68. 
69. ## Decision
70. 
71. Accept unrestricted outbound network access. The agent's security model relies on:
72. 
73. 1. **Credential scoping** — PAT has only Copilot Requests; App has only PR write + contents read

76. 4. **Runtime hardening** — `cap_drop: ALL`, `no-new-privileges`, non-root, resource limits
77. 5. **No host access** — no Docker socket, no host mounts, no personal credentials
78. 
79. Even with egress filtering, data could be exfiltrated through allowed endpoints (create gists, post to issues). The realistic threat is prompt injection via malicious PR/issue content, and the mitigations above address that directly.
80. 
81. ### Outcome
82. 
83. Agent network isolation is **closed as won't-fix**. The analysis shows egress filtering is either technically infeasible (port publishing constraint), fragile (IP-based rules), or insufficient (proxy bypass, DNS exfil). The existing controls provide the actual security boundary.

1. # ADR-008: Persistent Worktree Retention for Agent Tasks
2. 
3. **Date:** 2026-04-11
4. **Status:** Accepted
5. 
6. ## Context
7. 
8. The agent service now drives both code review and issue implementation flows. The
9. original worktree lifecycle optimised for cleanup only:
10. 
11. 1. `/reviews` was not a named Docker volume, so container restarts deleted every
12.    worktree, Copilot transcript, raw review JSON file, and `.copilot/` session
13.    directory.
14. 2. `cleanup_worktree()` and `cleanup_branch_worktree()` removed worktrees
15.    immediately in `finally` blocks, leaving no debugging window after a task
16.    finished.
17. 3. The unified implement lifecycle needs Copilot CLI `--resume`, which depends on
18.    the `.copilot/` directory surviving across multiple CLI invocations in the same
19.    retained worktree.
20. 
21. We need to preserve the latest task artifacts long enough to debug failures and
22. to let fix loops reuse the same Copilot session state, without letting `/reviews`
23. grow forever.
24. 
25. ## Options Considered
26. 
27. ### Option A: Keep eager deletion
28. 
29. Delete worktrees immediately after each task, as the original review flow did.
30. 
31. - ✅ Minimal disk usage
32. - ✅ No background cleanup logic
33. - ❌ Destroys `.copilot/` state, so `--resume` cannot work across retained tasks
34. - ❌ Removes raw artifacts needed for debugging failed or surprising runs
35. - **Verdict:** Rejected. It blocks the implement fix loop and leaves no debug window.
36. 
37. ### Option B: Persist `/reviews` and defer cleanup with retention
38. 
39. Store worktrees on a named volume, write an expiry marker on cleanup, and reap old
40. worktrees later.
41. 
42. - ✅ Preserves transcripts, raw review JSON, `.copilot/`, and checked-out code
43. - ✅ Enables `--resume` for the unified implement lifecycle
44. - ✅ Keeps cleanup bounded with a configurable retention period
45. - ❌ Requires marker parsing and a startup/background reaper
46. - ❌ Uses more disk than eager deletion
47. - **Verdict: Chosen.** Best balance of debuggability, resume support, and bounded storage.
48. 
49. ### Option C: Version every rerun (`pr-42-v1`, `pr-42-v2`, ...)
50. 
51. Keep every historical worktree instead of replacing the latest one.
52. 
53. - ✅ Full history for debugging across reruns
54. - ❌ More naming complexity and more disk growth
55. - ❌ Most debugging value is in the latest failed run, not every intermediate copy
56. - **Verdict:** Rejected. The extra history is not worth the operational cost right now.
57. 
58. ## Decision
59. 
60. Persist `/reviews` as a named volume and switch worktree cleanup from eager delete
61. to deferred cleanup:
62. 
63. 1. Worktree cleanup writes a `.cleanup-after` marker with an expiry timestamp and
64.    branch name instead of deleting immediately.
65. 2. `reap_old_worktrees()` scans `/reviews/` and removes worktrees whose marker has
66.    expired.
67. 3. The reaper runs on FastAPI startup and opportunistically before new worktree
68.    creation / cleanup operations.
69. 4. Retention defaults to 14 days via `WORKTREE_RETENTION_SECONDS`, overridable by
70.    environment variable.
71. 5. Re-reviews and reruns still use **replace semantics**: if `pr-42` or
72.    `agent/issue-59` already exists, remove that worktree first and recreate it from
73.    the latest remote state.
